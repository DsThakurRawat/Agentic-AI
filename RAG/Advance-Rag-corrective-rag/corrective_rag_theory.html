Module 1: The Core Limitations of Standard RAG

Traditional Retrieval-Augmented Generation (RAG) is built on a massive, highly idealistic assumption: **that vector databases retrieve correct, relevant contexts 100% of the time**.

In real-world deployment, this assumption falls apart. Standard RAG retrievers routinely retrieve irrelevant, stale, or incomplete information due to two critical issues:

  • ❌ Semantic Gap Failure: Embedding algorithms measure lexical and semantic similarity, not factual truth. A document that matches the query’s vocabulary perfectly can actually be completely irrelevant or contain false statements.
  • ❌ Context Window Noise: Packing full chunked documents into the LLM's context window exposes the generator to irrelevant background noise, leading to "Lost in the Middle" syndrome and semantic drift.

"When an LLM generator is fed irrelevant or incorrect context, it either produces garbage answers or hallucinates, completely defeating the purpose of the RAG system."

Standard RAG Lifecycle (Slide 1 Overview)

No validation occurs. Whatever semantic similarity returns is directly injected into the generator prompt.

Module 2: Corrective RAG (CRAG) Foundations

Introduced in early 2024, **Corrective Retrieval-Augmented Generation (CRAG)** resolves these flaws by implementing a lightweight, active **Retrieval Evaluator** node.

Instead of sending retrieved contexts straight to the generator, the Evaluator intercepts the flow. It assesses the relevance of the retrieved documents against the user query, assigns a confidence grade, and dynamically branches execution into three correction workflows:

Route 1: Correct (Confidence ≥ 0.8)

The vector documents match the query's criteria perfectly. CRAG forwards the documents to a **Knowledge Refinement** stage to strip away noise and stitch only high-signal sentences for generation.

Route 2: Incorrect (Confidence < 0.2)

The vector documents are completely useless. CRAG bypasses the database completely, rewrites the query, and triggers concurrent **Web Search (Tavily search)** to retrieve external ground-truth facts.

Route 3: Ambiguous (0.2 ≤ Score < 0.8)

The vector documents contain some useful context but are incomplete. CRAG branches concurrently: refining internal contexts ($k_{in}$) AND querying web engines ($k_{ex}$), blending both sets of context.

Module 3: Core Flowchart & Routing Mechanics

The Routing Decisions

The core decision-making within CRAG is controlled by the **Retrieval Evaluator**. By prompting the evaluation LLM to output a clean JSON structured schema containing a confidence probability, the system determines the next stage:

Score ≥ 0.8: => Correct (Knowledge Refinement)
Score < 0.2: => Incorrect (Tavily Web Search)
0.2 ≤ Score < 0.8: => Ambiguous (Concurrently Blends Both)

This exact 3-stage conditional branching matches the **Iteration 2 Flowchart** (from Slide 3). It ensures that no matter what the vector search provides, the agent contains a recovery buffer to rescue the generation phase!

Module 4: Knowledge Refinement (Slide 2 Focus)

In standard RAG, chunked documents (typically 500-1000 tokens) are fed raw into the context window. This includes menus, repetitive terminology, and unrelated background sentences.

CRAG resolves this through **Knowledge Refinement** (conceptualized in your Slide 2 drawing). When a document $d_i$ is flagged as containing relevant information, it undergoes a surgical three-step deconstruction:

  • 1. Decompose: The chunked document $d_i$ is decomposed into a list of fine-grained sentence strips: $[s_1, s_2, ..., s_k]$.
  • 2. Filter: A lightweight evaluation model (originally a T5 model with 770M parameters, or structured prompt rules in modern LLMs) evaluates each individual sentence strip. It discards strips that do not directly address the query $x$.
  • 3. Recompose: The high-signal sentence strips that pass the relevance filter are stitched back together into a clean, highly condensed context block ($k_{in}$).

Sentence-Strip Deconstruction Pipeline

Sentence strips are evaluated in isolation. Discarding noise yields highly concentrated, relevant facts for the generation model.

Module 5: Dynamic Query Rewriting & Web Searching

When internal data is graded as **Incorrect** or **Ambiguous**, CRAG activates the **Knowledge Searching** route (Slide 4). Simply throwing the user's conversational query straight into a search engine is ineffective; search engines expect concise search tokens, not conversational dialogue.

To extract high-signal answers from the web, the system operates in two stages:

  • 1. Query Rewriter Node: An LLM rewrites the complex user request $x$ into 3-5 distinct, targeted search engine queries optimized for Google or Tavily (e.g., rewriting "Who was the screenwriter for Death of a Batman?" into "Death of a Batman screenwriter", "Death of a Batman writer credits").
  • 2. Web Search Refinement Node: The queries are executed in parallel on Tavily Search API. The raw web search logs are parsed, key URLs are extracted, and snippets are compiled. The system then selects the most relevant fragments and stores them as external knowledge context ($k_{ex}$).

Query Re-engineering & Search Ingestion

# Query Rewrite Blueprint
REWRITE_SYSTEM = """You are a search query re-engineer.
Rewrite the complex user query into 3 concise search engine queries.
Keep queries short, keyword-focused, and stripped of conversational fluff.
"""

def query_rewrite_node(state: State) -> dict:
    rewriter = llm.with_structured_output(SearchQueries)
    result = rewriter.invoke([
        SystemMessage(content=REWRITE_SYSTEM),
        HumanMessage(content=f"Query: {state['query']}")
    ])
    return {"search_queries": result.queries}
Translating raw dialogue into keyword-based search vectors ensures Tavily retrieves exact factual matches.

Module 6: Blended Context Synthesis (Ambiguous Mode)

The real magic of Corrective RAG lies in its handling of **Ambiguity**. Often, retrieved document databases are neither completely correct nor completely incorrect. They might contain part of an answer but lack crucial modern context, or cite a fact without listing its recent updates.

When the Evaluator grades the retrieved context in the ambiguous zone ($0.2 \le \text{Score} < 0.8$), CRAG refuses to settle for partial information or risk web hallucination. It triggers a **Blended Context Synthesis** pathway:

1. Refine Vectors
Extracts clean internal facts (k_in)
+
2. Search Web
Extracts external ground truth (k_ex)
=>
3. Blend & Stitch
Injects both into Generator prompt

By weaving internal proprietary documents ($k_{in}$) together with fresh web facts ($k_{ex}$), CRAG gives the generator LLM a comprehensive, ground-truth foundation. The resulting response is highly detailed, contextually complete, and verified on all fronts!

Notebooks 1 & 2: Embedding & Database Lookup

# In database.py
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings
from langchain_text_splitters import RecursiveCharacterTextSplitter

def initialize_vector_db(documents_list):
    # Strip text blocks and chunk securely
    splitter = RecursiveCharacterTextSplitter(chunk_size=600, chunk_overlap=100)
    chunks = splitter.split_documents(documents_list)
    
    # Ingest chunks into Chroma database
    db = Chroma.from_documents(chunks, OpenAIEmbeddings(model="text-embedding-3-small"))
    return db.as_retriever(search_kwargs={"k": 4})

def retrieve_context(state: State) -> dict:
    retriever = state["retriever"]
    docs = retriever.invoke(state["query"])
    return {"retrieved_docs": docs}

Notebook 3: Structuring Evaluator JSON Outputs

class EvaluationGrade(BaseModel):
    confidence_score: float = Field(..., description="Relevance score from 0.0 to 1.0.")
    rationale: str = Field(..., description="Brief logical justification for this grade.")

EVAL_SYSTEM = """You are a rigorous information quality auditor.
Analyze the retrieved document context against the user query.
Assign a confidence score from 0.0 (completely irrelevant) to 1.0 (perfect factual answer).
"""

def evaluate_retrieval_node(state: State) -> dict:
    grader = llm.with_structured_output(EvaluationGrade)
    docs = state["retrieved_docs"]
    query = state["query"]
    
    # Score the accumulated documents
    grade = grader.invoke([
        SystemMessage(content=EVAL_SYSTEM),
        HumanMessage(content=f"Query: {query}\nRetrieved: {docs}")
    ])
    
    # Establish dynamic routing boundaries
    if grade.confidence_score >= 0.8:
        internal_grade = "correct"
    elif grade.confidence_score < 0.2:
        internal_grade = "incorrect"
    else:
        internal_grade = "ambiguous"
        
    return {
        "eval_score": grade.confidence_score,
        "internal_grade": internal_grade
    }

Notebooks 4 & 5: Sentence Strip Decomposition

def refine_knowledge_node(state: State) -> dict:
    docs = state["retrieved_docs"]
    query = state["query"]
    refined_strips = []
    
    # Step 1: Decompose document into sentences
    for doc in docs:
        sentences = doc.page_content.split(". ")
        
        # Step 2: Evaluate sentence strips in isolation
        for sent in sentences:
            if not sent.strip():
                continue
            is_relevant = check_sentence_relevance(sent, query) # LLM classifier
            
            # Step 3: Filter & Accumulate
            if is_relevant:
                refined_strips.append(sent.strip())
                
    # Step 4: Recompose into clean context (k_in)
    k_in = ". ".join(refined_strips)
    return {"refined_context_k_in": k_in}

Notebook 6: LangGraph Compilation & Routes

def routing_edge(state: State) -> str:
    # Grade-based conditional router edge
    grade = state["internal_grade"]
    if grade == "correct":
        return "refine"
    elif grade == "incorrect":
        return "rewrite"
    else:
        return "ambiguous"

# Building the Active Graph Schema
workflow = StateGraph(State)
workflow.add_node("retrieve", retrieve_context)
workflow.add_node("evaluator", evaluate_retrieval_node)
workflow.add_node("refine", refine_knowledge_node)
workflow.add_node("rewrite", query_rewrite_node)
workflow.add_node("web_search", web_search_node)
workflow.add_node("generator", generator_synthesis_node)

workflow.add_edge(START, "retrieve")
workflow.add_edge("retrieve", "evaluator")

# Branch dynamically based on Evaluator confidence
workflow.add_conditional_edges(
    "evaluator",
    routing_edge,
    {
        "refine": "refine",
        "rewrite": "rewrite",
        "ambiguous": "refine" # In ambiguous, trigger refine + rewrite concurrently
    }
)
# Connect paths to dynamic generator
workflow.add_edge("refine", "generator")
workflow.add_edge("rewrite", "web_search")
workflow.add_edge("web_search", "generator")
workflow.add_edge("generator", END)

crag_app = workflow.compile()
