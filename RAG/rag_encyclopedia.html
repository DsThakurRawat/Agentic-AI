The Anatomy of
Modern RAG Systems

RAG isn't just a tool; it's an architecture for **Dynamic Grounding**. It solves the "Stale Knowledge" problem by providing LLMs with an external, searchable memory.

00 The Science of Retrieval

Parametric vs. Non-Parametric

Parametric Knowledge: This is the knowledge stored in the LLM's weights (parameters). It's like a person's long-term memory. It's fast but gets outdated quickly and is hard to correct.

Non-Parametric Knowledge: This is the external data (PDFs, Databases, Web) that RAG provides. It's like a person's library or notebook. It's infinitely scalable and always up-to-date.

In-Context Learning (ICL)

The miracle of RAG is based on In-Context Learning. This is the ability of an LLM to "learn" or adapt to new information provided within the prompt itself, without any permanent changes to its weights. We are essentially using the LLM's attention mechanism to focus on the provided context.

RAG essentially shifts the burden of **Factual Accuracy** from the model's weights to the retrieval engine, allowing the model to focus purely on its core strength: Reasoning and Synthesis.

01 Solving the Three Pillars of Failure

1. Knowledge Cutoff

Models are frozen in time (e.g., Oct 2023). RAG allows them to answer questions about events that happened 5 minutes ago by fetching fresh news or internal docs.

2. Data Privacy

Uploading private data to OpenAI for fine-tuning is a security risk. RAG keeps your data in your local Vector DB (like FAISS) and only sends the relevant snippets.

3. Hallucination Control

When an LLM doesn't know an answer, it guesses. In RAG, we tell the model: "Use ONLY the provided context. If it's not there, say you don't know."

01. Ingestion
The Splitter

LLMs have finite attention. We use Recursive Splitters to break documents into chunks that preserve semantic boundaries (paragraphs, sentences) rather than just character counts.

02. Embeddings
Vector Space

We transform text into Tensor Embeddings. This projects words into a multi-dimensional map where "King" and "Queen" are mathematically closer than "King" and "Apple".

03. Retrieval
Semantic Search

Unlike SQL (Exact Match), we use K-Nearest Neighbors (KNN) to find the most meaningful context, even if the user uses different words than the document.

04. Synthesis
In-Context Logic

The LLM doesn't just "read" the context; it performs Attention-based Synthesis, cross-referencing the query with the retrieved snippets to produce a grounded response.

03 Implementation Walkthrough

1. The Naive RAG Script

def implement_naive_rag(documents, query):
    # 1. Recursive splitting to preserve context
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=500, chunk_overlap=50
    )
    texts = splitter.create_documents(documents)
    
    # 2. Local vector storage (FAISS)
    vs = FAISS.from_documents(texts, embeddings)
    retriever = vs.as_retriever()
    
    # 3. Augmenting the prompt
    chain = (
        {"context": retriever, "question": RunnablePassthrough()}
        | prompt | llm | StrOutputParser()
    )
    return chain.invoke(query)
How this code works:
  • ➜ The RecursiveCharacterTextSplitter ensures that paragraphs aren't split mid-sentence.
  • ➜ FAISS (Facebook AI Similarity Search) builds an efficient index for fast retrieval.
  • ➜ The RunnablePassthrough allows the question to flow directly into the prompt alongside the retrieved context.

2. The Agentic (LangGraph) Pattern

Why use LangGraph for RAG?

Naive RAG is linear. Agentic RAG is **Cyclical**. In the code below, we don't just "Retrieve and Generate"—we define a state machine that can grade its own performance.

  • Stateful Routing: Decide if a web search is needed based on local retrieval scores.
  • Self-Correction: If the LLM generates an answer not found in the context, re-run the loop.
