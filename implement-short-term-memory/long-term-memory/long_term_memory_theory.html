Section 1: The Memory Taxonomy: Episodic vs. Semantic

In autonomous agent engineering, a major architectural pitfall is treating conversation history as the sole memory source. When a thread ends, or is reset due to context limits, the agent's memory of that user disappears. This is **transient episodic memory**.

"Without a cross-thread persistence layer, starting a 'New Chat' forces the agent to re-introduce itself, re-learn user preferences, and repeat context collection loops."

To achieve persistent personalization, we split memory into two core subsystems:

  • Episodic (Checkpointer) – Tied directly to a thread_id. Saves the linear message log, active execution checkpoints, and temporary node states. Access is isolated strictly within the single active thread.
  • Semantic (BaseStore) – Tied to an open-ended, structured namespace (e.g. ("users", user_id)). Persistent and completely cross-thread, allowing the agent to immediately inherit context across distinct conversation lifetimes.

Hierarchical Namespacing

Namespaces inside `BaseStore` act as isolated folders using **tuples of strings** to group keys. This layout lets you segment memories at granular organizational layers (e.g. scoping data to specific organizations, individual users, or unique tasks).

Example Namespace Structures:

  • 📁 ("users", user_id)
    Isolates specific user preferences, bio, and style guides.
  • 📁 ("orgs", org_id, "policies")
    Enforces shared compliance checks across an entire company.

Core CRUD Interface

The `BaseStore` provides an extremely simple upsert interface. Whether backed by a mock `InMemoryStore` during unit testing, or a persistent `PostgresStore` in production, your agent logic relies on identical semantic calls.

API Methods Blueprint:

  • store.put(ns, key, dict_val) – Atomic upsert.
  • store.get(ns, key) – Atomic fetch.
  • store.search(ns_prefix, query) – Vector similarity lookup.
Extensible: SQLiteStore, PostgresStore, InMemoryStore

Deep-Dive: Reflective Summarization & Fact Duplication

Directly storing raw user messages leads to index contamination. If a user states their favorite coding language is Python multiple times, a naive database write will create duplicate fact keys.

To prevent this, our architecture deploys a background **Memory Extraction Node** triggered asynchronously at step boundaries.

Structured Profile Schema

from pydantic import BaseModel, Field

class UserProfile(BaseModel):
    name: str = Field(description="User's preferred name")
    interests: List[str] = Field(default_factory=list, description="Extracted coding/personal interests")
    custom_rules: List[str] = Field(default_factory=list, description="Style guidelines or rules explicitly set")

The extraction node uses llm.with_structured_output(UserProfile), fetching the existing profile from store.get(), merging newly stated facts with current records, and rewriting a clean, deduplicated profile back to the Postgres Store. This keeps the database clean and context windows small!

Reflective Extraction Lifecycle

Step-by-Step Operations:
  1. The user chats inside the active thread.
  2. The extraction node triggers and reads the existing profile from the store namespace.
  3. The LLM merges newly stated preferences with the existing Pydantic schema.
  4. The deduplicated object upserts back to Postgres, making it instantly available to other threads!

PostgresStore Initialization Blueprint

from langgraph.store.postgres import PostgresStore
from langgraph.checkpoint.postgres import PostgresSaver

# Establish active database pool
pool = ConnectionPool(conninfo="postgresql://user:pass@localhost:5432/dbname")

# BaseStore persistent semantic storage
store = PostgresStore(pool)
store.setup()  # Set up relational schema tables

# Checkpointer episodic state storage
checkpointer = PostgresSaver(pool)
checkpointer.setup()

# Compile Master Graph with both memory modules
app = workflow.compile(checkpointer=checkpointer, store=store)

Semantic Background Extractor Node

def extract_memories(state: State, config: RunnableConfig):
    store = config["configurable"]["store"]
    user_id = config["configurable"]["user_id"]
    namespace = ("users", user_id)

    # 1. Fetch current profile
    current = store.get(namespace, "profile")
    profile = current.value if current else {"interests": [], "custom_rules": []}

    # 2. Extract facts from recent dialogue
    extractor = llm.with_structured_output(UserProfile)
    raw_extraction = extractor.invoke([
        SystemMessage(content=EXTRACTION_SYSTEM),
        HumanMessage(content=f"Recent Messages: {state['messages'][-5:]}")
    ])

    # 3. Merge Pydantic structures & Deduplicate
    updated_interests = list(set(profile["interests"] + raw_extraction.interests))
    updated_rules = list(set(profile["custom_rules"] + raw_extraction.custom_rules))

    # 4. Save upsert back to PostgresStore
    store.put(namespace, "profile", {
        "interests": updated_interests,
        "custom_rules": updated_rules
    })

LTM Hydrator Node

def hydrate_ltm_context(state: State, config: RunnableConfig):
    store = config["configurable"]["store"]
    user_id = config["configurable"]["user_id"]
    namespace = ("users", user_id)

    # Fetch LTM Profile
    record = store.get(namespace, "profile")
    if record:
        profile_content = (
            f"User Interests: {record.value['interests']}\n"
            f"User Guidelines: {record.value['custom_rules']}"
        )
    else:
        profile_content = "No long-term profile active."

    # Inject LTM context directly into system instructions
    system_instruction = SYSTEM_BASE.format(ltm_context=profile_content)
    return {"system_instruction": system_instruction}
