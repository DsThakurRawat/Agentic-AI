Subgraph Orchestration

Break the monolith. Build complex agentic systems by encapsulating logic into independent, state-isolated sub-graphs that communicate via clean interfaces.

Theory Why Subgraphs are Needed?

In traditional software development, we don't write one 10,000-line function. We use modules, classes, and microservices. **Subgraphs are the Microservices of Agentic AI.**

Modularity

Divide a massive problem into bite-sized, independent workflows.

Reusability

Build a "RAG Subgraph" once and use it across ten different parent agents.

Failure Isolation

If the "Coding Subgraph" fails, the "Testing Subgraph" can retry it without crashing the whole system.

State Separation

Each subgraph has its own private state, preventing "Global Variable Hell".

The Two Orchestration Patterns

Pattern 01

Independent Subgraphs (Bridge Pattern)

Use this when the subgraph has a completely different state schema. This is perfect for utility graphs (like Translation, Summarization, or RAG) that shouldn't know anything about your application's domain logic.

  • Explicit Mapping: You must manually translate parent keys to subgraph keys.

  • Encapsulation: The subgraph is treated as an external black box.

# Parent Node acts as a Bridge
def translate_node(state: ParentState):
    # 1. Map Parent -> Subgraph
    sub_input = {"input_text": state["ans_en"]}
    
    # 2. Invoke Subgraph manually
    res = subgraph.invoke(sub_input)
    
    # 3. Map Subgraph -> Parent
    return {"ans_hi": res["translated_text"]}
Pattern 02

Shared State (Direct Integration)

Use this when the subgraph shares keys with the parent state. The subgraph is added directly to the parent graph as a node. LangGraph automatically manages the state flow.

  • Native Flow: Updates returned by the subgraph merge directly into parent state.

  • Reduced Boilerplate: No need for manual bridge functions.

# Add compiled graph directly as a node
parent_builder.add_node("answer", generate_node)
parent_builder.add_node("translate", subgraph)

parent_builder.add_edge(START, "answer")
parent_builder.add_edge("answer", "translate")
parent_builder.add_edge("translate", END)

03 Failure Isolation & Retries

One of the most powerful reasons to use subgraphs is Isolation. In a monolithic graph, a failure in a specific logic branch can be hard to recover from without resetting the entire flow.

Production Tip:

Apply RetryPolicies specifically to the Subgraph nodes. If your "Research Subgraph" fails due to a rate limit, the Parent orchestrator can catch the error and decide to skip the node or try an alternative data source, keeping the rest of the application alive.

graph TD P[Parent Graph] --> S[Subgraph: Coding] S -- "Exception" --> Catch{Error Handler} Catch -- "Retry Subgraph" --> S Catch -- "Degraded Mode" --> Final[Parent Result]

Observability in the Hierarchy

When streaming from a parent graph, subgraph events are hidden by default. To "look inside" the nested execution, use the subgraphs=True flag. 

for chunk in graph.stream(
    {"input": "..."} , 
    subgraphs=True, 
    stream_mode="updates"
):
    print(chunk) # Streams internal subgraph nodes too!
