01. The Context Window

Think of the Context Window as the LLM's **Working Memory**. It's the maximum amount of information (tokens) the model can "see" and reason over at any single moment.

The Camera Analogy

A model with a small context window is like a **narrow-angle lens**—it can see the detail of a single flower but loses the forest. A model with a large context window is a **wide-angle lens**, capturing the entire mountain range of data.

The Context Collapse Problem

Trimming

The "Blunt & Efficient" approach. When the conversation exceeds the limit, we simply delete the oldest messages.

  • Fast execution
  • No token waste
  • Loses early context

Summarization

The "Intelligent & Lossy" approach. Condensing a massive thread into a single paragraph summary.

  • Retains core intent
  • Flexible memory
  • LLM cost per summary

Hybrid Context

The "Modern Standard". Combining a Summary of past context with the most recent 10-20 messages for precision.

  • Best of both worlds
  • production-ready

02. Long-Term Memory (LTM)

While Short-Term memory lives in the prompt, Long-Term memory lives in **Databases**. This is information that defines **continuity** across sessions, days, and weeks.

1

Episodic Memory

"What Happened"

Past events and experiences tied to specific time/context. Useful for tracking the state of long-running tasks.

2

Semantic Memory

"What is True"

Stable facts about the user or the world. "The user is a Senior Developer", "Budget is $5000".

3

Procedural Memory

"How to do it"

Strategies, rules, and learned behaviors. "When solving SQL, avoid subqueries".

Persistence Mechanics

Checkpointers

The "Save Game" feature of LangGraph. Automatically saves state to SQLite/Postgres after every node execution.

Thread IDs

The key to isolation. thread_id="user_123" ensures that the AI remembers THIS user's context specifically.

# Enable Persistence
checkpointer = SqliteSaver.from_conn_string(":memory:")
graph = builder.compile(checkpointer=checkpointer)

# Invoke with context

The 4-Step Memory Pipeline

1. Creation / Update

The system analyzes the latest turn: "Is anything here worth remembering beyond this conversation?". It extracts **Memory Candidates** and filters out noise.

01
Extract: "User prefers aisle seats."
Filter: Noise deleted (e.g. greetings).

2. Selective Storage

Memories are persisted. Episodic events go to Structured DBs; Semantic knowledge goes to **Vector Databases** for semantic lookup.

02
Store: Vector Store (ChromaDB/Pinecone)
Index: Semantic Embeddings created.

3. Precision Retrieval

"Given current situation, what do I need now?". The system performs a **Semantic Search** to pull only relevant context, avoiding prompt bloat.

03
Search: Cosine Similarity lookup.
Result: "Seat preference" retrieved.

4. Injection (Influencing LLM)

The retrieved memory is inserted into the **Short-Term Memory (STM)**. The LLM sees it as tokens in the prompt, allowing it to behave with continuity.

04
Inject: Prompt Prefix updated.
Result: Coherent, context-aware response.

Implementation Patterns: From Basics to Advanced

Pattern A: Clean Deletion (Reducers)

Using RemoveMessage to tell the state reducer to permanently delete specific tokens, keeping the state lean.

from langchain.messages import RemoveMessage

def cleanup_node(state):
    msgs = state["messages"]
    if len(msgs) > 10:
        # Delete the oldest 6 messages
        return {"messages": [RemoveMessage(id=m.id) for m in msgs[:6]]}

Pattern B: Hybrid Summarization

Recursive summarization maintained in the state. Retains the "gist" of the conversation indefinitely.

def summarize(state):
    summary = state.get("summary", "")
    res = llm.invoke(f"Update summary: {summary}")
    return {"summary": res.content, "messages": state["messages"][-2:]}
