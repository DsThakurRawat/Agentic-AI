The Evolution of Agency: Reactive vs. Planning Agents

Traditional AI agents act **reactively**. In a standard loop (like ReAct), the agent observes the user query, selects a tool, runs the tool, and immediately generates an intermediate response. While highly effective for simple questions, this reactive structure breaks down under complex, multi-step, or structured requirements (such as writing a cohesive 3,000-word system design document or writing complex APIs).

"Without a structural outline, reactive agents suffer from cognitive drift. They forget their high-level objective, repeat tasks redundantly, and produce uncoordinated, disjointed output."

A **Planning Agent** resolves this failure by decoupling the reasoning process into two distinct, structured phases:

  • Phase 1: Planning / Deconstruction – The LLM acts as an architect, generating a structured, validated, and comprehensive plan (outline) composed of independent tasks before executing any tools or writing content.
  • Phase 2: Execution / Assembly – Specialized modules execute the planned steps. The plan is either executed sequentially with dynamic adjustments (Plan-and-Execute) or dispatched in parallel to multiple instances (Orchestrator-Worker).

Architecture A: Plan-and-Execute

Inspired by **BabyAGI** and **Plan-and-Solve** methodologies, this architecture is heavily **sequential** and **dynamic**. The agent creates a plan consisting of steps, runs the first step, and then routes back to a *Replanner* node. The Replanner inspects the state, marks the step completed, makes adjustments to the remaining steps, and triggers execution of the next step.

Plan-and-Execute Core Properties:

  • High Task Interdependence: Step B requires output from Step A.
  • Re-planning Feedback Loop: Dynamic outline restructuring at each superstep.
  • Low Thread Concurrency: Executed strictly one step at a time.
Best For: Deep Research, Autonomous Coding Agents, Self-Correction Loops

Architecture B: Orchestrator-Worker

This architecture relies on **parallel task deconstruction**. An *Orchestrator* analyzes the objective and spits out an array of independent tasks. Using LangGraph's dynamic `Send` API, the system forks execution, launching specialized *Worker* nodes for each task concurrently. Finally, a *Reducer* node aggregates all parallel results.

Orchestrator-Worker Core Properties:

  • Low Task Interdependence: Subtasks can be executed in isolation.
  • Maximum Concurrency: Parallel fan-out yields massive speed boosts.
  • State Aggregation Engine: Relies on Reducers to combine state values.
Best For: Multi-Section Blogs, Parallel Web Scraping, Code Auditing Swarms

Deep-Dive: State Reducers & The Value Clobbering Problem

When executing nodes in parallel (Fan-out), LangGraph runs workers concurrently. If multiple worker nodes try to write to the same key in the Graph's global state, standard state machines will overwrite each other's outputs. This is known as **value clobbering**.

LangGraph resolves this elegantly using **Reducers**. A Reducer is a function associated with a specific key in the `TypedDict` schema. When a node returns a value for that key, LangGraph runs the Reducer function rather than replacing the value.

The Code Archetype

# In state.py
from typing import Annotated, TypedDict
import operator

class State(TypedDict):
    topic: str
    # The Reducer: operator.add appends new items to the existing list
    sections: Annotated[List[tuple[int, str]], operator.add]
    final: str

In the **Blog Writing Agent** implemented in your bwa_backend.py, the sections are stored as tuples of `(task_id, section_markdown)`. The reducer combines these tuples into an unordered list. When the execution flows into the **Reducer Node (`merge_content`)**, it sorts the list by `task_id` (index 0) to ensure sections are assembled in the precise planned order, solving parallel synchronization seamlessly!

Section 3.5: The Dual Architectural Functions of Reducers

In high-concurrency LLM agents, a Reducer performs two distinct functions: one at the State Schema level (data collection) and one at the Graph Compilation level (subgraph pipeline processing).

1. State-Level Reducers: Accumulation without Clobbering

When a graph spawns multiple parallel worker nodes (fan-out), they execute concurrently in isolated threads. When they complete, they publish their drafts to a shared state key (sections). Without a reducer, each thread's write would overwrite (clobber) the previous ones, resulting in a single final section surviving.

By declaring sections: Annotated[List[tuple[int, str]], operator.add] in our state contract, we assign a **State-Level Reducer**. Instead of replacing the value, the LangGraph runtime automatically appends every worker's (task_id, section_markdown) payload as it arrives, accumulating all drafts into an unordered list in global state memory.

2. Graph-Level Reducers: Compiled Subgraph Pipeline

Collecting the drafts is only the first step. To transition this raw list into a professional, cohesive asset, the reduction phase is elevated into a nested Reducer Subgraph (compiled separately as a sub-agent).

This sub-agent performs a strict three-stage execution flow:

  • Stitching (merge_content): Sorts the accumulated tuples by `task_id` to guarantee the sections are assembled in the precise ordered blueprint regardless of which worker finished first.
  • Asset Speculation (decide_images): An LLM scans the text and dynamically injects visual indicators (e.g. [[IMAGE_1]]) mapping to structured ImageSpec targets (placeholder, filename, caption, prompt, size).
  • Dynamic Compilation (generate_and_place_images): Dispatches visual prompts to multimodal image models (Gemini-2.5) and search-replaces the placeholders with final styled image tags.

Section 4: Strategic Upfront Routing & Topic Profiling

In reactive multi-agent flows, search queries are usually fired on-demand as workers stumble upon gaps in their local knowledge. While convenient, this causes massive execution overhead, redundancy, and unpredictability.

A robust Planning Agent solves this by implementing a **Strategic Upfront Router**. Before any outlines are drafted, the user query is classified by a routing LLM node using llm.with_structured_output(RouterDecision). The router maps the query into one of three strict operating profiles:

  • 1. Closed Book: Evergreen academic or theoretical concepts. No research is needed, saving API calls and reducing execution time to a bare minimum.
  • 2. Hybrid: Evergreen concepts that benefit from modern, live technical examples or software version updates. Spawns 3–5 targeted search queries.
  • 3. Open Book: Time-sensitive weekly roundups, active corporate announcements, release updates, or pricing reviews. Spawns up to 10 search queries.

Section 5: Concurrent Grounded Research & Deduplication

Once a topic is identified as requiring external research (either `hybrid` or `open_book`), the router synthesizes a sequence of up to ten highly specific search queries. If we were to send these queries sequentially, search times would stack lineally, blocking the agent pipeline.

Instead, the Research Node executes search queries concurrently (using Tavily Search APIs). However, concurrent searching across related queries introduces two critical engineering issues:

  1. State Memory Bloat: Raw search logs contain vast amounts of boilerplate HTML code, tracking parameters, and generic layout text.
  2. Citation Overlap: Different queries will inevitably hit identical primary URLs, creating messy, duplicate citations in the compiled bibliography.

To solve this, the agent processes raw outputs using a specialized schema extractor. It filters evidence into clean, unified EvidenceItem data models and maps them into a global dictionary keyed by the URL. This strictly ensures that each source is cited **exactly once** in the compiled EvidencePack.

Deduplication & Extraction Engine

def research_node(state: State) -> dict:
    queries = state.get("queries") or []
    raw = []
    # Concurrently execute queries in parallel
    for q in queries:
        raw.extend(_tavily_search(q, max_results=6))
    
    if not raw:
        return {"evidence": []}

    # Restructure using structured extraction LLM
    extractor = llm.with_structured_output(EvidencePack)
    pack = extractor.invoke([
        SystemMessage(content=RESEARCH_SYSTEM),
        HumanMessage(content=f"Raw Results: {raw}")
    ])

    # Strict URL-key deduplication
    dedup = {}
    for e in pack.evidence:
        if e.url:
            dedup[e.url] = e
            
    evidence = list(dedup.values())
    return {"evidence": evidence}
By mapping by e.url inside our dedup dictionary, the context window receives only clean, unique information payloads.

Section 6: Temporal Grounding & Hard Recency Filtering

LLMs live in a "temporal vacuum". They do not inherently know the current year or date unless it is injected. When asked about modern tech developments, their internal weights skew towards historic data, causing them to hallucinate active facts (e.g., asserting that an old beta version is still the active build).

To prevent this "time-travel" issue, the Planning Agent introduces strict **Contextual As-Of Controls**:

  • ⏱️ State-Bound Epoch: The graph state maintains an as_of string representing the exact simulated datetime of the execution.
  • ⏱️ Dynamic Recency Cutoff: The router assigns a recency_days threshold. For weekly news roundups, this is locked to 7 days. For hybrid configurations, it extends to 45 days.
  • ⏱️ Hard Time Filter: Inside the research node, the date of search articles is extracted and normalized to ISO YYYY-MM-DD. Any item published older than as_of - recency_days is immediately stripped from the state.

Section 7: Mid-Blog Cognitive Drift Guardrails

In a classic parallel map-reduce flow, each worker operates in an isolated environment, unaware of what other workers are doing. In blog writing, this leads to **Cognitive Drift**: a worker tasked with writing a section on "Latest MCP Releases" might drift into writing setup guides, code blocks, or installation tutorials, causing frustrating repetition and a disjointed overall flow.

We enforce absolute outline cohesion using a three-tier guardrail system:

  • Explicit Task Contracts: Each section is strictly defined by an active Pydantic Task model including precise target word counts, specific bullet subtopics, and true/false flags for requires_code or requires_citations.
  • Global Kind Constraints: The orchestrator passes a global blog_kind value (e.g. news_roundup) to every worker. In a news roundup, worker system prompts strictly prohibit any coding tutorials, keeping the focus strictly on events and market implications.
  • Citation Scope Locks: Under open_book mode, workers are explicitly blocked from introducing any claim, model, or funding update unless supported by an active evidence URL. Claims must have an explicit citation link matching [Source](URL).

Section 8: Nested Reducer Subgraphs & Multi-Modal Generation

Stitching parallel sections together is a basic reduction. But professional publishing pipelines require more than raw markdown text; they need tailored **multimodal visual assets** to illustrate complex topics.

To solve this without cluttering the main blog writing state machine, we structure the final reduction phase as an independent, nested **Reducer Subgraph** (compiled using its own state rules):

  1. Stitch & Sort (merge_content): Workers push their draft snippets into the global state as (task_id, section_markdown). The Reducer sorts these by the original task ID and merges them, preserving structural outline cohesion.
  2. Asset Speculation (decide_images): An editing LLM scans the full stitched document. It decides if technical illustrations are needed (max 3) and injects literal markers like [[IMAGE_1]] directly into the text. It outputs a GlobalImagePlan featuring custom visual prompt plans and filename targets.
  3. Multimodal Ingestion (generate_and_place_images): The system loops through the planned specifications. It sends the visual prompts to Google's gemini-2.5-flash-image model, downloads the compiled bytes as PNGs under images/, and search-replaces the markdown placeholders with fully rendered image tags.

"Failsafe Engineering: If the multimodal API returns a safety cutoff, quota limit, or network block, the graph catches the exception and prints an interactive, graceful fallback quote box describing what the diagram would illustrate."

LangGraph Implementation Blueprint

Review the robust and structured layout of your planning agent. Underneath the hood, this agent uses Pydantic structured output mapping for strict verification, multi-threaded mapping for speed, and a nested Reducer Subgraph to execute clean modular steps.

Pydantic Schemas (Pydantic V2)

class Task(BaseModel):
    id: int
    title: str
    goal: str = Field(..., description="One sentence describing what the reader should understand.")
    bullets: List[str] = Field(..., min_length=3, max_length=6)
    target_words: int = Field(..., description="Target words (120–550).")
    tags: List[str] = Field(default_factory=list)
    requires_research: bool = False
    requires_citations: bool = False
    requires_code: bool = False

class Plan(BaseModel):
    blog_title: str
    audience: str
    tone: str
    blog_kind: Literal["explainer", "tutorial", "news_roundup", "comparison", "system_design"] = "explainer"
    constraints: List[str] = Field(default_factory=list)
    tasks: List[Task]

class State(TypedDict):
    topic: str
    mode: str  # closed_book, hybrid, open_book
    needs_research: bool
    queries: List[str]
    evidence: List[EvidenceItem]
    plan: Optional[Plan]
    sections: Annotated[List[tuple[int, str]], operator.add]  # Reducer field!
    merged_md: str
    final: str

Routing & Tavily Engine Nodes

def router_node(state: State) -> dict:
    decider = llm.with_structured_output(RouterDecision)
    decision = decider.invoke(
        [
            SystemMessage(content=ROUTER_SYSTEM),
            HumanMessage(content=f"Topic: {state['topic']}\nAs-of date: {state['as_of']}"),
        ]
    )
    return {
        "needs_research": decision.needs_research,
        "mode": decision.mode,
        "queries": decision.queries,
    }

def route_next(state: State) -> str:
    # Routing Edge decides next state node path
    return "research" if state["needs_research"] else "orchestrator"

def research_node(state: State) -> dict:
    queries = state.get("queries") or []
    raw = []
    for q in queries:
        raw.extend(_tavily_search(q))  # Invokes Tavily engine
    
    extractor = llm.with_structured_output(EvidencePack)
    pack = extractor.invoke([SystemMessage(content=RESEARCH_SYSTEM), ...])
    return {"evidence": pack.evidence}

Orchestration & Dynamic Send API

def orchestrator_node(state: State) -> dict:
    planner = llm.with_structured_output(Plan)
    plan = planner.invoke([
        SystemMessage(content=ORCH_SYSTEM),
        HumanMessage(content=f"Topic: {state['topic']}\nEvidence:\n{state['evidence']}")
    ])
    return {"plan": plan}

def fanout(state: State):
    # LangGraph Send API: Dynamically triggers worker threads in parallel
    assert state["plan"] is not None
    return [
        Send(
            "worker",
            {
                "task": task.model_dump(),
                "topic": state["topic"],
                "evidence": [e.model_dump() for e in state.get("evidence", [])],
                "plan": state["plan"].model_dump(),
            },
        )
        for task in state["plan"].tasks
    ]

Parallel Workers & Reducer Subgraph compiling

def worker_node(payload: dict) -> dict:
    # Standard isolated drafting node
    task = Task(**payload["task"])
    section_md = llm.invoke([
        SystemMessage(content=WORKER_SYSTEM),
        HumanMessage(content=f"Draft section: {task.title}\nGoal: {task.goal}")
    ]).content
    return {"sections": [(task.id, section_md)]}

def merge_content(state: State) -> dict:
    # Sort by original task ID to secure structured order
    ordered = [md for _, md in sorted(state["sections"], key=lambda x: x[0])]
    merged_md = "\n\n".join(ordered)
    return {"merged_md": merged_md}
